## Context

### 当前状态

github-buddy 的 IP 检测模块（`internal/detector/`）使用三维度检测：
1. **ICMP ping**：获取延迟和丢包率
2. **TCP 443 端口**：检测 HTTPS 端口连通性（`net.DialContext("tcp", ip:443)`）
3. **TCP 22 端口**：检测 SSH 端口连通性

评分公式为：`延迟(50%) + 丢包率(30%) + 端口连通(20%)`

### 问题

在国内网络环境下，**TCP 连接成功 ≠ HTTPS 真正可用**：
- GFW 的 DNS 污染可能将域名解析到错误 IP，该 IP 的 443 端口可能仍然开放
- 中间人设备可能响应 TCP SYN，但 TLS 握手失败（证书不匹配）
- GitHub CDN IP 频繁变动，某 IP 的 443 端口虽开放，但已不再为目标域名提供服务
- 结果：`update` 命令选出的 IP 实际不可用，用户体验极差

### 约束
- 无需引入新的第三方依赖，Go 标准库 `crypto/tls` + `net/http` 即可
- 总检测超时保持 30 秒上限
- 向后兼容，不移除已有的 TCP/ICMP 检测

## Goals / Non-Goals

**Goals:**
- 通过 HTTPS 应用层验证确保选出的 IP 真正可用于 GitHub 访问
- HTTPS 验证结果在评分中占主导权重，未通过验证的 IP 大幅降分
- 保持检测流程的并发性和性能
- 区分不同域名的 HTTPS 验证路径（如 github.com 用 `/`，api.github.com 用 `/`）

**Non-Goals:**
- 不实现完整的 HTTP 内容校验（不检查页面内容是否正确）
- 不实现 HTTP/2 或 QUIC 协议检测
- 不改变数据源获取逻辑
- 不修改 hosts 文件写入逻辑

## Decisions

### 决策 1：HTTPS 验证方式

**选择**：使用 `crypto/tls` 直接进行 TLS 握手验证，而非完整 HTTP 请求

**原因**：
- TLS 握手是验证 IP 是否真正服务于目标域名的最关键步骤
- TLS 握手验证 = 确认服务端持有目标域名的有效证书 = IP 真正可用
- 相比完整 HTTP 请求，TLS 握手更轻量、更快
- 如果需要进一步验证，可选做一次简单的 HTTP HEAD 请求

**备选方案**：
- 完整 HTTP GET 请求：更重，但能验证 HTTP 层面的可用性。作为第二阶段增强考虑
- 仅检查证书域名：过于简单，无法确认完整 TLS 链路

### 决策 2：HTTPS 验证超时

**选择**：单次 HTTPS 验证超时设为 5 秒

**原因**：
- TLS 握手通常在 1-3 秒内完成
- 5 秒足以覆盖大部分正常场景
- 所有候选 IP 的 HTTPS 验证并发执行，不影响总超时

### 决策 3：评分权重重新分配

**选择**：新的评分权重体系

| 维度 | 旧权重 | 新权重 | 说明 |
|------|--------|--------|------|
| HTTPS 验证 | 无 | 40% | 新增，最关键的可用性指标 |
| 延迟 | 50% | 30% | 降低，但仍然重要 |
| 丢包率 | 30% | 15% | 降低 |
| 端口连通 | 20% | 15% | 降低 |

**原因**：
- HTTPS 验证通过是 IP 可用的充要条件，必须占最高权重
- 在 HTTPS 都通过的前提下，延迟是区分优劣的关键
- HTTPS 未通过的 IP，无论其他维度多优秀，总分都会很低

### 决策 4：HTTPS 验证的具体实现

**选择**：对候选 IP 执行 TLS 握手 + 简单 HTTP HEAD 请求

实现流程：
```
1. 创建自定义 tls.Config，设置 ServerName 为目标域名
2. 创建自定义 http.Transport，通过 DialTLSContext 指定连接目标 IP:443
3. 发起 HTTP HEAD 请求（Host: 目标域名）
4. 验证：TLS 握手成功 + 状态码 < 500
5. 记录 HTTPS 延迟（含 TLS 握手 + HTTP 往返）
```

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|---------|
| HTTPS 验证增加检测耗时 | 与 TCP/ICMP 检测并发执行，单个 IP 超时 5 秒 |
| 某些域名可能不支持 HEAD 请求 | 降级为 GET 请求，且不读取 body |
| TLS 指纹被 GFW 识别 | 使用 Go 默认 TLS 配置，不添加异常指纹 |
| 所有 IP 的 HTTPS 验证都失败 | 回退到 TCP-only 评分模式，避免选不出任何 IP |
| 增加对 GitHub 服务器的请求量 | 使用 HEAD 请求最小化开销，且候选 IP 通常不超过 10 个 |
